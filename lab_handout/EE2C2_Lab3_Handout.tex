\section{Topic 0 --- Getting started: Vivado, the design flow, and the project}

\subsection{Background}
A \textbf{Field-Programmable Gate Array (FPGA)} is a chip full of configurable look-up tables,
flip-flops, wiring, and I/O. You describe hardware in \textbf{SystemVerilog}; the tool configures
the FPGA to \emph{become} that hardware. \textbf{Vivado} is the design tool. The path from source
to a running board is the \textbf{design flow}:

\begin{lstlisting}[style=plain]
SystemVerilog (RTL) --simulate--> Synthesis (RTL -> LUT/FF netlist)
   --> Implementation (place & route) --> Bitstream (.bit) --> Program over JTAG
\end{lstlisting}
You will walk the tool flow in Topics 1--6 using an \textbf{AC701 / Artix-7} project, because that
board and device support are installed on the university PCs by default. The physical lab board on the
bench is the \textbf{RealDigital AUP-ZU3-4GB}; the AUP-ZU3 demos in Topics 7--8 are programmed from
\textbf{prebuilt bitstreams} so you can still see the LED behaviour even if the campus Vivado installation
does not include XCZU3EG synthesis/implementation support.

\section{Topic 1 --- Combinational logic}

\subsection{Background}
\textbf{Combinational logic} produces outputs that depend \emph{only on the current inputs}; it has
no memory of the past. Gates, multiplexers, decoders, and comparators are all combinational. Because there
is no clock and no stored state, the output is not ``computed once'' like a line of software; it is a piece
of hardware that is \emph{always present} and continuously reflects its inputs. When an input changes, the
output settles to its new value after a short \textbf{propagation delay} through the gates.

In SystemVerilog you describe combinational logic either with a continuous \texttt{assign} or inside an
\texttt{always\_comb} block, and both build the same kind of hardware:
\begin{lstlisting}[style=sv]
assign y = (a & b) | (~a & c);   // continuous assignment

always_comb begin                // procedural form, still combinational
    y = (a & b) | (~a & c);
end
\end{lstlisting}

The lab module \texttt{src/comb\_led\_logic.sv} is a single \texttt{always\_comb} block that drives eight
LED bits, each a different function of the four switches \texttt{sw[3:0]} and the button \texttt{btn}. Its
SystemVerilog operators (\texttt{\&}, \texttt{|}, \texttt{\textasciicircum}, \texttt{\textasciitilde}, the
\texttt{? :} conditional, and the reduction \texttt{\&sw}) each map directly onto a piece of logic hardware. Working out which is which is the goal of Checkpoint~1a.

\begin{notebox}
\textbf{Avoid latches.} Inside an \texttt{always\_comb} block, every output must be assigned on every path
through the block. If a bit is left unassigned for some input combination, the synthesiser has to
``remember'' its previous value and infers an unwanted memory element, a \textbf{latch}. The lab code
prevents this by writing a default \texttt{led\_comb = 8'b0000\_0000;} as its first line, so every bit
always has a defined value.
\end{notebox}

\section{Topic 2 --- Sequential logic and FSM}

\subsection{Background}
\textbf{Sequential logic} has memory: outputs can depend on past inputs. State is stored in
\textbf{flip-flops} that update only on a \textbf{clock edge}.
\begin{itemize}[noitemsep]
  \item \texttt{always\_ff @(posedge clk)} describes flip-flops.
  \item \texttt{<=} (non-blocking) is used for clocked updates.
  \item A \textbf{counter} is the simplest example.
\end{itemize}
\begin{lstlisting}[style=sv]
always_ff @(posedge clk) begin
    if (rst) count <= '0;
    else     count <= count + 1'b1;
end
\end{lstlisting}
The lab module \texttt{src/one\_hz\_tick.sv} counts clock cycles and, on the wrap cycle, raises
\texttt{tick\_1s} for exactly one clock cycle. In the AC701 implementation, \texttt{src/top.sv}
passes \texttt{CLK\_FREQ\_HZ = 200\_000\_000} so the counter wraps once per second on the 200\,MHz
board clock.

\begin{notebox}
\textbf{A tick is a clock \emph{enable}, not a new clock.} \texttt{tick\_1s} is a normal signal that
is high for one cycle. The whole design still runs on the single 200\,MHz \texttt{clk}; \texttt{tick\_1s}
just marks the one cycle on which the FSM is allowed to advance.
\end{notebox}

A \textbf{finite state machine (FSM)} is built directly on this sequential logic: it moves through defined
\textbf{states} and has three parts: a \textbf{state register} (flip-flops, \emph{sequential}),
\textbf{next-state logic} (\emph{combinational}), and \textbf{output decode logic} (\emph{combinational}).
The lab FSM (\texttt{src/led\_fsm.sv}) walks a five-state ring, lighting one of the AC701's four user
LEDs at a time and then turning them all off:
\begin{lstlisting}[style=plain]
IDLE -> MONO_LED_0 -> MONO_LED_1 -> MONO_LED_2 -> ALL_OFF -> (back to IDLE)
\end{lstlisting}
A transition happens \emph{only when} \texttt{tick\_1s} is high, which is once per second on the board. So the
counter you just met is what \emph{paces} the FSM: \texttt{tick\_1s} is the clock-enable that lets the
state register advance one step, without adding a second clock.

\section{Topic 4 --- Static timing analysis}

\subsection{Background}
The AC701 board clock is 200\,MHz, so the clock period is 5\,ns. Every signal launched by one flip-flop on a
clock edge must travel through the combinational logic, reach the next flip-flop, and \emph{settle} before
the following edge captures it. \textbf{Static timing analysis} (STA) checks this for \emph{every}
register-to-register path in the design at once. Unlike the simulations in Topics 1--2, STA uses no input
vectors; it works purely from the netlist delays and the clock constraint, so it proves timing for
\emph{all} input combinations, not just the ones a testbench happens to drive.

\paragraph{Setup constraint (data must not arrive too late).}
Start from the one-period bound and add a margin $t_{\mathrm{unc}}$ for \textbf{clock uncertainty} (clock
jitter plus skew between the launch and capture flip-flops):
\[
  T_{\mathrm{clk}} \;\ge\; t_{\mathrm{cq}} + t_{\mathrm{comb}} + t_{\mathrm{su}} + t_{\mathrm{unc}}.
\]
This uses the \emph{slowest} (longest) combinational path, so that path sets the maximum usable clock
frequency. The \textbf{setup slack} is how much room is left over:
\[
  t_{\mathrm{slack}} \;=\; T_{\mathrm{clk}} - \bigl(t_{\mathrm{cq}} + t_{\mathrm{comb}} + t_{\mathrm{su}}
  + t_{\mathrm{unc}}\bigr).
\]
Negative slack means the path is too slow to meet the clock period, and the captured value is undefined.

\paragraph{Hold constraint (data must not arrive too early).}
Hold checks the same path in its \emph{fastest} version, using the minimum delays $t_{\mathrm{cq,min}}$ and
$t_{\mathrm{comb,min}}$ (setup used the slowest):
\[
  t_{\mathrm{cq,min}} + t_{\mathrm{comb,min}} \;\ge\; t_{\mathrm{h}} + t_{\mathrm{unc}}.
\]
The corresponding \textbf{hold slack} is
\[
  t_{\mathrm{slack}} \;=\; \bigl(t_{\mathrm{cq,min}} + t_{\mathrm{comb,min}}\bigr)
  - \bigl(t_{\mathrm{h}} + t_{\mathrm{unc}}\bigr).
\]
The shortest path must be long enough that the newly launched value does not race through and overwrite the
data before the capturing flip-flop has latched the \emph{previous} value. This is a short-path problem,
checked separately from setup. Because the inequality contains no $T_{\mathrm{clk}}$ term, a hold violation
\textbf{cannot be fixed by slowing the clock}.

\paragraph{What Vivado reports.}
A path with positive slack meets timing with room to spare; negative slack means it fails. Vivado summarises
this across the whole design with:
\begin{itemize}[noitemsep]
  \item \textbf{WNS} (Worst Negative Slack) is the slack of the single tightest
        setup path. $\mathrm{WNS} \ge 0$ means \emph{every} setup path is met.
  \item \textbf{TNS} (Total Negative Slack) is the sum of all negative setup
        slacks. $\mathrm{TNS} = 0$ confirms there are no setup violations.
  \item \textbf{WHS / THS} are the Worst / Total Hold Slack, the hold-side
        equivalents of WNS / TNS.
  \item The \textbf{critical path} is the path with the smallest slack, i.e.\ the
        one closest to failing (and the first to fix if timing is not met).
\end{itemize}
In this small design the critical path almost always lies inside the one-second counter in
\texttt{one\_hz\_tick}. With \texttt{CLK\_FREQ\_HZ = 200\,000\,000} the counter is about 28 bits wide, and
its increment adder and terminal-count comparator form the longest combinational chain between two
flip-flops; everything else (the FSM, the LED multiplexer) is far shorter.

\begin{notebox}
The switches and reset button are slow, human-driven inputs, so \texttt{ac701\_lab.xdc} marks them
\texttt{set\_false\_path}. STA therefore ignores paths that start at those pins and reports only the internal
register-to-register paths, which is why the critical path lives inside the counter rather than at an input.
\end{notebox}

\section{Topic 5 --- Power analysis}

\subsection{Background}
Vivado's power report \emph{estimates} on-chip power and splits it into two parts:
\textbf{static} power (transistor leakage, present whenever the device is powered)
and \textbf{dynamic} power (drawn only when nodes switch). Vivado further breaks the
dynamic part into \textbf{clock}, \textbf{logic/signal}, and \textbf{I/O} power.

Dynamic switching power follows
\[
  P_{\mathrm{dyn}} \;\approx\; \alpha\, C\, V_{\mathrm{DD}}^{2}\, f,
\]
where $\alpha$ is the switching (toggle) activity, $C$ the switched capacitance,
$V_{\mathrm{DD}}$ the supply voltage, and $f$ the clock frequency. The activity
$\alpha$ is the key unknown: unless you supply real switching data (e.g.\ a SAIF
file from simulation), Vivado performs a \emph{vectorless} estimate using assumed
default toggle rates, and flags this with a low \nav{Confidence level}. The
$V_{\mathrm{DD}}^{2}$ term is why supply voltage dominates the energy budget, and
the linear $f$ term is why the clock and clock-driven logic are often the largest
dynamic contributors.

You read this report on the \textbf{AC701} implementation from Topic 3, running at 200\,MHz. Because the
design is tiny and its only continuously toggling logic is the free-running counter inside
\texttt{one\_hz\_tick}, the clock network and that counter dominate the dynamic estimate, while everything
else contributes very little.

\section{Topic 7 --- PWM DAC: the digital/analog boundary}

\subsection{Background}
A digital output pin can only be 0 or 1, yet we often want an \emph{analog} level, for example a
dimmable LED. \textbf{Pulse-Width Modulation (PWM)} fakes one by switching the pin on and off very fast and
varying the \textbf{fraction} of each period that it stays high. That fraction is the \textbf{duty cycle}.
The LED (and your eye) average the fast square wave, so the perceived brightness follows the duty cycle.

PWM is built from two pieces you have already met: a \textbf{sequential} counter and a \textbf{combinational}
comparator. A free-running $N$-bit counter ramps $0, 1, 2, \ldots, 2^N{-}1$ and wraps; on every clock the
output is
\[ \texttt{pwm\_out} = (\texttt{count} < \texttt{duty}). \]
So \texttt{pwm\_out} is high for the first \texttt{duty} counts of each $2^N$-cycle period and low for the
rest, and its average value, and therefore the LED brightness, is \texttt{duty}$/2^N$. The waveform below
shows how the comparator turns the counter ramp into a pulse train:

    \draw[tudelft, very thick] (0,0)--(4,2)--(4,0)--(8,2)--(8,0)--(9,1);
    \draw[orange, very thick, dashed] (0,1.3)--(9,1.3);
